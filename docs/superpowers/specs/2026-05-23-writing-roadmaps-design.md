# Writing Roadmaps Skill Design

## Goal

Add a `writing-roadmaps` skill that sits optionally between `brainstorming` and `writing-plans` for large designs that do not fit comfortably into a single detailed implementation plan or context window.

## Background

`pi-supergsd` packages patched Superpowers skills for Pi. Most skills are generated by `npm run updater` from upstream Superpowers plus declarative patches in `updater/skills/*.json`.

Upstream Superpowers does not currently include a roadmap-writing skill. This project will add `writing-roadmaps` as a local committed skill under `skills/`, not as an updater-managed upstream skill.

The existing runtime in `index.ts` already serves all directories under `skills/`, so no runtime code change is needed.

## User-Approved Approach

Use a local `writing-roadmaps` skill plus stronger cross-skill workflow contracts:

1. `brainstorming` chooses either `writing-roadmaps` or `writing-plans` after an approved design.
2. `writing-roadmaps` creates one high-level roadmap document with ordered phases.
3. `writing-plans` writes one detailed implementation plan for exactly one roadmap phase at a time.

This enables large work to be split across multiple planning/execution contexts while preserving coherent phase boundaries.

## New Skill: `writing-roadmaps`

### Location

Create:

- `skills/writing-roadmaps/SKILL.md`

This skill is local-only. It will not have an `updater/skills/writing-roadmaps.json` definition.

### Trigger

Use after an approved design/spec when the work is too large for one detailed implementation plan, is clearly multi-phase, or may exceed one context window.

### Roadmap Output

Roadmaps should be saved to:

- `docs/superpowers/roadmaps/YYYY-MM-DD-<feature>-roadmap.md`

User preferences for roadmap location override this default.

### Roadmap Content

The roadmap document should include:

- Full feature goal
- Link/path to the approved design spec
- Planning strategy explaining why a roadmap is needed
- Ordered phases starting at Phase 1
- For each phase:
  - outcome
  - why this phase comes now
  - included scope
  - explicitly deferred/out-of-scope work
  - likely affected files or areas
  - dependencies
  - verification criteria
  - risks and mitigations
  - context-window notes if relevant

### Phase Rules

Plans should always start from Phase 1 and then proceed sequentially unless the user explicitly changes the order.

There should be no hard phase-count limit. The skill should prefer as few phases as make sense, but allow more than five phases when needed to keep each phase independently plannable and context-sized.

Every phase must leave the project in a sensible intermediate state:

- functionality not broken
- tests/CI expected green
- no half-migrations or dangling integrations
- later phases may add capability, but earlier phases must remain coherent on their own

### What Roadmaps Must Not Do

Roadmaps must not become detailed implementation plans. They should avoid:

- task-level checkbox steps
- full code blocks
- detailed TDD step sequences
- implementation minutiae better suited for `writing-plans`

### Handoff

After saving a roadmap, the skill should direct the agent to use `/skill:writing-plans` for Phase 1 first.

## Brainstorming Skill Patch

Update `updater/skills/brainstorming.json` so generated `skills/brainstorming/SKILL.md` no longer treats `writing-plans` as the only next step.

Required generated behavior:

- Checklist terminal step chooses `/skill:writing-roadmaps` for large, multi-phase, or context-heavy designs, otherwise `/skill:writing-plans`.
- Process flow terminal node changes from invoking writing-plans to choosing between writing-roadmaps and writing-plans.
- Terminal-state guidance says:
  - use `/skill:writing-roadmaps` when the design is too large for one detailed plan/context window
  - use `/skill:writing-plans` when the design fits in one detailed plan
- The post-design handoff message should ask the user to review the spec before choosing the next planning step, not before starting an implementation plan.

## Writing-Plans Skill Patch

Update `updater/skills/writing-plans.json` so generated `skills/writing-plans/SKILL.md` is roadmap-aware.

Required generated behavior:

- Description includes selected roadmap phase as a trigger.
- Plan save path includes phase-plan naming:
  - `docs/superpowers/plans/YYYY-MM-DD-<feature>-phase-N-<phase-name>.md`
- Scope check says:
  - if a roadmap exists, write exactly one detailed plan for one selected phase
  - default target is Phase 1 or the next sequential unplanned phase
  - do not combine phases unless the user explicitly instructs it
  - if the target phase is unclear, ask before planning
- Plan header includes:
  - roadmap path or `None`
  - phase name or `Single-plan implementation`
- Self-review checks:
  - selected phase coverage
  - no leakage of later phases into the current plan
  - executing this phase should leave the project functional with tests/CI green
- Execution handoff says:
  - execution should stop at the phase boundary
  - future phases require separate detailed plans

## Files to Change

Implementation should touch only:

- `skills/writing-roadmaps/SKILL.md`
- `updater/skills/brainstorming.json`
- `updater/skills/writing-plans.json`
- generated `skills/brainstorming/SKILL.md`
- generated `skills/writing-plans/SKILL.md`

No `index.ts` changes are required.

## Verification Plan

Run:

```bash
npm run updater
npm test
```

Also review generated skill text manually to confirm the intended roadmap language appears in `skills/brainstorming/SKILL.md` and `skills/writing-plans/SKILL.md`.

For TypeScript, run an updater-only type-check if full `npx tsc --noEmit` is blocked by missing Pi runtime types for `index.ts`, as noted in `AGENTS.md`.

## Out of Scope

- Making `writing-roadmaps` updater-managed
- Updating `executing-plans` or branch-finishing skills
- Changing runtime skill discovery
- Adding automated tests for skill prose
